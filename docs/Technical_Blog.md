# 构建高性能电力系统进程间数据交换中心：IPCSharedDataPool 设计与实践

> 在电力监控、DCS控制等工业场景中，多进程架构因其高可靠性和故障隔离能力被广泛采用。然而，进程间如何高效共享实时数据一直是个难题。本文介绍 IPCSharedDataPool 的设计思路与实现方案。

---

## 一、背景：为什么需要跨进程数据池

### 1.1 多进程架构的必然选择

在电力监控系统、变电站自动化、DCS 分布式控制等关键基础设施领域，系统可靠性是第一要务。传统的单体程序存在致命缺陷：一旦某个模块崩溃，整个系统都将瘫痪。

多进程架构因此成为首选：

```
┌─────────────────────────────────────────────────────┐
│                    监控主机                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ 通信进程  │  │ 业务进程  │  │  UI进程   │          │
│  │ (独立崩溃)│  │ (独立崩溃)│  │ (独立崩溃)│          │
│  └──────────┘  └──────────┘  └──────────┘          │
│       ↓              ↓              ↓               │
│   规约解析崩溃    逻辑处理崩溃    界面卡死           │
│   不影响其他      不影响其他      不影响其他          │
└─────────────────────────────────────────────────────┘
```

**优势**：进程隔离，故障不传播，可用性大幅提升。

### 1.2 进程间通信的困境

多进程架构带来了新的挑战：**进程间如何高效交换数据？**

传统方案对比：

| 方案 | 延迟 | 吞吐量 | 复杂度 | 适用场景 |
|------|:----:|:------:|:------:|----------|
| 消息队列 | 毫秒级 | 中 | 低 | 低频控制命令 |
| Socket TCP | 毫秒级 | 中 | 高 | 跨主机通信 |
| 管道 | 微秒级 | 低 | 低 | 单向数据流 |
| 共享内存 | **纳秒级** | **极高** | 中 | **高频实时数据** |

对于电力监控系统：
- 遥测数据每秒更新数千次
- 遥信变位要求毫秒级响应
- 百万级数据点需要并发访问

**结论**：共享内存是唯一能满足性能要求的方案。

### 2.3 共享内存布局

采用 SoA（Structure of Arrays）布局，提升缓存命中率：

```
共享内存布局:
┌────────────────────────────────────────────────────────┐
│                    ShmHeader                           │
│  (魔数、版本、容量、进程计数、统计信息...)              │
├────────────────────────────────────────────────────────┤
│                    YX Data (遥信)                       │
│  ┌─────────────────────────────────────────────────┐  │
│  │ values[]     │ uint8_t 数组，紧凑存储            │  │
│  ├─────────────────────────────────────────────────┤  │
│  │ timestamps[] │ uint64_t 数组，毫秒时标           │  │
│  ├─────────────────────────────────────────────────┤  │
│  │ qualities[]  │ uint8_t 数组，质量码              │  │
│  └─────────────────────────────────────────────────┘  │
├────────────────────────────────────────────────────────┤
│                    YC Data (遥测)                       │
│  values[] | timestamps[] | qualities[]                 │
├────────────────────────────────────────────────────────┤
│                    Index Table                         │
│  哈希索引表：key → 索引映射                            │
├────────────────────────────────────────────────────────┤
│                    Process Info                        │
│  注册进程信息：PID、心跳时间、状态                     │
└────────────────────────────────────────────────────────┘
```

**为什么选择 SoA 而不是 AoS？**

```cpp
// AoS (Array of Structures) - 传统方式
struct DataPoint {
    uint8_t value;
    uint64_t timestamp;
    uint8_t quality;
};
DataPoint points[1000000];  // 访问 value 时加载整个结构体

// SoA (Structure of Arrays) - 本项目采用
struct DataPool {
    uint8_t values[1000000];      // 连续存储，缓存友好
    uint64_t timestamps[1000000]; // 批量访问时效率高
    uint8_t qualities[1000000];
};
```

当遍历所有数据点的值时，SoA 布局可减少 3 倍内存访问量。

## 三、核心组件实现

### 3.1 SharedDataPool：共享数据池

**设计要点**：

1. **按类型分区**：YX（遥信）、YC（遥测）、DZ（定值）、YK（遥控）独立存储
2. **两级索引**：64 位 key → 哈希表 → 数组索引，O(1) 访问
3. **读写分离**：读操作无锁，写操作加写锁

**关键代码**：

```cpp
// 点位 key 编码：设备地址 << 32 | 点号
inline uint64_t makeKey(uint32_t deviceAddr, uint32_t pointId) {
    return (static_cast<uint64_t>(deviceAddr) << 32) | pointId;
}

// 设置遥信值（纳秒级延迟）
bool setYX(uint64_t key, uint8_t value, uint8_t quality = 0) {
    uint32_t idx = findIndex(key);
    if (idx == INVALID_INDEX) return false;
    
    m_yxValues[idx] = value;
    m_yxTimestamps[idx] = getCurrentTime();
    m_yxQualities[idx] = quality;
    return true;
}

// 获取遥信值（无锁读取）
bool getYX(uint64_t key, uint8_t& value, uint8_t& quality) {
    uint32_t idx = findIndex(key);
    if (idx == INVALID_INDEX) return false;
    
    value = m_yxValues[idx];
    quality = m_yxQualities[idx];
    return true;
}
```

### 3.2 IPCEventCenter：事件中心

**设计要点**：

1. **环形缓冲区**：固定大小，覆盖最旧事件，防止内存溢出
2. **多订阅者**：每个订阅者独立消费位置，互不影响
3. **事件过滤**：支持按类型、按点位过滤

**事件结构**（固定 64 字节）：

```cpp
struct Event {
    uint64_t key;           // 数据点 key
    int32_t addr;           // 设备地址
    int32_t id;             // 点号
    PointType pointType;    // YX/YC/DZ/YK
    uint8_t reserved;       // 对齐填充
    
    union {
        uint32_t intValue;
        float floatValue;
    } oldValue, newValue;   // 变化前后值
    
    uint64_t timestamp;     // 事件时间戳
    uint8_t quality;        // 质量码
    uint8_t isCritical;     // 关键点标志
    uint16_t sourcePid;     // 来源进程 PID
    char source[16];        // 来源标识
};
```

**使用示例**：

```cpp
// 订阅事件
uint32_t subId = client->subscribe([](const Event& e) {
    printf("YX变化: addr=%d, id=%d, %d -> %d\n",
           e.addr, e.id, e.oldValue.intValue, e.newValue.intValue);
});

// 发布事件
client->publishEvent(key, PointType::YX, oldValue, newValue);

// 处理事件（在主循环中）
while (running) {
    client->processEvents(subId, 100);  // 最多处理 100 个
    usleep(10000);
}
```

### 3.3 SOERecorder：SOE 事件记录器

**SOE（Sequence of Events）** 是电力系统事故分析的关键数据，要求纳秒级时标精度。

**SOE 记录结构**（32 字节）：

```cpp
struct SOERecord {
    uint64_t absoluteTime;   // 绝对时标（纳秒）
    uint64_t monotonicTime;  // 单调时钟（纳秒）
    uint32_t pointKey;       // 点位标识
    uint16_t msoc;           // 毫秒计数器
    uint16_t sourcePid;      // 来源进程
    uint8_t pointType;       // 点位类型
    uint8_t eventType;       // 事件类型
    uint8_t quality;         // 质量码
    uint8_t oldValue;        // 旧值
    uint8_t newValue;        // 新值
    uint8_t priority;        // 优先级
};
```

**为什么需要双时标？**

- `absoluteTime`：实际时间，用于报表和事故记录
- `monotonicTime`：单调递增，不受系统时间调整影响，用于精确排序

### 3.4 VotingEngine：三取二表决引擎

电力系统中关键信号常采用三重冗余，需要表决逻辑确定最终值。

**表决策略**：

| 策略 | 说明 |
|------|------|
| 严格三取二 | 三源都有效，至少两源一致 |
| 宽松表决 | 允许一源无效，两源一致即可 |
| 优先级表决 | 按优先级顺序选择有效源 |

**表决配置**：

```cpp
struct VotingConfig {
    uint32_t groupId;        // 表决组 ID
    char name[32];           // 名称
    uint64_t sourceKeyA;     // 源 A 的 key
    uint64_t sourceKeyB;     // 源 B 的 key
    uint64_t sourceKeyC;     // 源 C 的 key
    uint8_t sourceType;      // 0=YX, 1=YC
    uint8_t votingStrategy;  // 表决策略
    float deviationLimit;    // 偏差限值
    uint32_t timeoutMs;      // 超时时间
};
```

**表决结果**：

```cpp
enum class VotingResult : uint8_t {
    UNANIMOUS = 0,     // 一致（三源相同）
    MAJORITY = 1,      // 多数（二取一）
    DISAGREE = 2,      // 不一致
    INSUFFICIENT = 3,  // 有效源不足
    FAILED = 4         // 表决失败
};
```

### 3.5 IEC61850Mapper：IEC 61850 数据映射

**IEC 61850** 是变电站自动化国际标准，定义了统一的数据模型。

**映射流程**：

```
SCL 配置文件 (ICD/SCD)

        ▼ pugixml 解析
┌───────────────────┐
│ IED / LD / LN /   │
│ DataSet / Report  │
└─────────┬─────────┘
          │
```

**关键映射**：

| IEC 61850 数据属性 | 映射类型 | 说明 |
|-------------------|----------|------|
| stVal (状态值) | YX | 遥信 |

| mag.i (整数测量) | YC | 遥测 |
| ctlVal (控制值) | YK | 遥控 |

## 六、设计经验总结

### 6.1 关键技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 共享内存实现 | POSIX shm | 跨进程安全，无文件依赖 |
| 同步机制 | pthread_rwlock | 支持跨进程，读写分离 |
| 内存布局 | SoA | 缓存友好，批量访问高效 |
| 事件存储 | 环形缓冲区 | 固定大小，无动态分配 |
| XML 解析 | pugixml | 轻量级，无外部依赖 |

### 6.2 踩过的坑

**坑 1：共享内存权限问题**

```cpp
// 错误：默认权限可能被其他进程拒绝
shm_open(name, O_CREAT | O_RDWR, 0666);

// 正确：显式设置权限
shm_open(name, O_CREAT | O_RDWR | O_EXCL, 0666);
fchmod(fd, 0666);  // 确保权限正确
```

**坑 2：进程崩溃后残留共享内存**

```cpp
// 解决方案：注册退出清理
atexit([]() {
    shm_unlink("/ipc_data_pool");
});

// 更好：记录 PID，重启时检测并清理
if (checkStaleProcess()) {
    cleanupStaleResources();
}
```

**坑 3：跨进程锁的优先级反转**

```cpp
// 问题：高优先级进程等待低优先级进程释放锁
// 解决：使用优先级继承属性
pthread_rwlockattr_setkind_np(&attr, PTHREAD_RWLOCK_PREFER_WRITER_NONRECURSIVE_NP);
```

## 七、总结

IPCSharedDataPool 通过共享内存 + 事件中心的架构，为电力/工业控制领域的多进程软件提供了高性能、高可靠的数据交换基础设施。

**核心价值**：

- ⚡ **性能**：纳秒级延迟，百万级吞吐
- 🛡️ **可靠**：进程隔离，故障不传播
- 🔌 **易用**：简洁 API，零学习成本
- 📐 **标准**：IEC 61850 数据模型支持

这套方案已在实际项目中验证，支撑了变电站监控、DCS 控制等多个关键系统的高可靠运行。
